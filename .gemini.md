# Pehlix App - Gemini Agent Context File

This file contains crucial architectural decisions, patterns, and bugfix notes that should be followed when working on this repository.

## Frontend Patterns

### 1. Dropdowns and Combo Boxes
- **Shadcn UI `Select`:** Radix UI `Select` strictly requires that every option has a valid `value`. **Never use `value=""`**. Instead, use a placeholder value like `value="none"` and then sanitize it before sending the payload to the backend (e.g., if `value === 'none'`, delete the key or send `undefined`).
- **Dynamic Combo Boxes (Creating on the fly):** When a dropdown needs to support selecting an existing item OR typing a new string to instantly create a new item (e.g., Doctors and Agents), do NOT use standard Shadcn `Select`. Instead, use `react-select/creatable` (`CreatableSelect`).

### 2. Data Fetching & Array Mapping (`@tanstack/react-query`)
- **API Response Structures:** Due to nested response wrappers from the backend (e.g., `{ status: 'success', data: { visits: [], total: ... } }`), always be extremely careful when destructuring or mapping over data. 
- **Precedence in Fallbacks:** When falling back to alternative object structures, wrap your array checks in parentheses. For example:
  ```javascript
  // CORRECT
  const patients = patientsData?.visits || patientsData?.patients || (Array.isArray(patientsData) ? patientsData : []);
  
  // INCORRECT (Ternary operator precedence issue will cause a crash)
  const patients = patientsData?.visits || patientsData?.patients || Array.isArray(patientsData) ? patientsData : [];
  ```

## Backend Patterns

### 1. Doctors & Agents Commission Tracking
- **Revenue & Referrals Tracking:** The `totalReferrals` and `totalRevenue` fields on the `Doctor` model must be updated **exactly once** when the visit is initially created. This logic resides in `VisitService.createVisit`.
- **Commission Calculation (`DoctorService.calculateAndRecordCommission`):** This function is called when a visit is fully paid (which can happen at registration if paid by cash, or later via `PaymentService` if the patient pays online). To prevent double-counting, this function **must not** increment `totalReferrals` and `totalRevenue`. It should strictly only generate the `Commission` record (with idempotency checks).
- **Zero Commission Handling:** If an agent has `commissionType === 'none'` or `commissionValue === 0`, they still get their `totalReferrals` and `totalRevenue` tracked, but the commission calculation function will simply return `null` and generate no commission record.

### 2. Staff Hybrid Multi-Role System
- **Roles vs Display Title:** The system uses a hybrid approach for staff management. A user has a string array `roles: [String]` (which dictates actual system permissions like `receptionist`, `technician`) and a `jobTitle: String` (which is purely for display on the frontend, like "Senior Pathologist").
- **Phased Migration & Fallback:** To prevent breaking existing production users, the `User` model currently retains the legacy `role` field. When checking permissions, **always check both**. In the frontend, the check is usually `user.roles?.includes('technician') || user.role === 'technician'`. In the backend middleware, `req.user.roles.includes(...)` handles it safely.
- **Middleware Usage:** Never use `req.user.role === ...` in controllers. Always use `req.user.roles.includes(...)`. The `requireRole` middleware handles checking the arrays securely.

### 3. Pathologist Approval Queue
- **Queue Preview Cards:** In `app/(lab)/reports/page.jsx`, the Approval Queue cards must extract and display key `parameters` inline directly on the card so pathologists have immediate context on what values they are approving (e.g. rendering `Hemoglobin: 13 g/dL`) without having to click the modal.

### 4. Billing & Payments Management
- **Invoice Balance Due:** `invoice.balance` is rarely populated directly on the raw document. Always manually calculate the balance in frontend components via `(invoice.totalAmount || 0) - (invoice.amountPaid || 0)`.
- **Payment Deletion / Adjustments:** We have two workflows to correct billing mistakes:
  1. **Delete a Payment:** Deleting a `Payment` automatically deducts `payment.amount` from `invoice.amountPaid` and drops the status to `partial` or `pending`.
  2. **Edit Invoice (Override):** Direct modification of `paymentStatus` and `amountPaid` on the invoice is allowed via `updateInvoice`. If the amount paid exactly matches the total amount, the backend safely autocorrects the status to `paid`.

### 5. PDF Generation & Margins (Report Service)
- **Margin Units:** When interacting with Puppeteer `page.pdf` margins, always append unit string identifiers (`'mm'`, `'px'`) to the values provided by the settings. Raw numbers will default to pixels and misalign on print.
- **Letterhead vs Full Background:** The `reportSettings.backgroundMode` dictates how the background image is injected. In `full_page` mode, we must inject a full background base64 covering `<body>` and `html`. Crucially, when `full_page` mode is enabled, the custom repeating table header must NOT be rendered, because the full background image already typically contains the header artwork.
